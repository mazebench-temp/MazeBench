## Subsystem: movement

MazeBench movement/pushing lives in /Users/jpappas/code/GitHub/PixelGameTest/public/maze-engine.js. A turn is engine.move(state,dx,dy,options) (7065): it builds an occupied Set keyed "x,y,elev" (buildOccupiedMap 2093), orders movers front-most-first along the axis (sortPlayersForMove 7000: dx>0 → x desc, tie y asc, tie lowest elevation; one representative per clone group), then resolves each player sequentially against mutating state with snapshot/rollback. Each player steps in a loop (7289): ice slopes resolve via resolveIceSlopeTraversal (2292), holes only enterable at elevation 0 (7434), standing requires terrain or actor support minus main players (canPlayerStandAtElevation 1692), blockers are any non-collectible actor at the exact target elevation (blockingActorAtElevation 2960). Pushes are allowed only on the first step (isInitialStep, 7479) — sliding players can't push boxes (but can push slope blockers via pushSlopeBlocker 7336). Push budget = countSupportingPlayers (2934): 1 + contiguous trailing players at the pusher's elevation; pushWeightForType (75) charges 1 for box/floating_floor and 0 for everything else including weightless_box; attemptPushActor (4588) charges cost, recursively pushes pushable blockers, and fails on: board edge, insufficient budget, non-pushable blocker (players/clones aren't pushable), or — heavy boxes only — no terrain support at the target elevation (4737; actor support doesn't count, so heavy boxes can't be pushed off ledges or onto stacks). moveBox (2832) slides via findSlideDestination (2655) across ice, then lands via landingElevationAtLocation (1719), which may drop several levels and can land on player heads. floating_floor is a weight-1 box that fills holes: applyHoleFalls (6062-6080) marks fillsHole, applyMoveFinalState (6136) turns the cell to floor. M0–M4 are NOT weights — level_parsing.json:130 maps all five tokens to weightless_box, differing only in groupId; group members keep fixed relative elevations (333), push as rigidly-expanding clusters (collectWeightlessPushCluster 3749 merges groups hit in the push direction; other pushables become recursive blockers), step with no support requirement (weightlessMemberCanOccupy 3114), settle one layer at a time (3293), slide while any member touches ice and none is supported (3214), can't leave the board, and can't laterally enter the side of floor/ice slabs (terrainSupportSideBlocksWeightlessEntry 731). Post-move, vertical-contact components (2232) fall/remove as a unit inside syncDynamicActorElevationsAndFalls (6145). Clones move as clusters of type "clone" with budget = member count (7120) and carry riders standing exactly +1 (cloneRidersForMove 1173); carried players skip their own move (7080). No swap/pass-through exists: gems/buttons/punchers are non-blocking; everything else blocks at exact elevation, and head-on movers see each other. Post passes: collapse moves, punchers (bounded loops), hole falls, lift toggles, dynamic elevation sync, final state (7985-8073). play-movement.js rebuilds an engine per keypress from runtime state (explicit elevations) and maps engine move records to animation/undo.

### Player move attempt (engine.move)

move(state, dx, dy, options) at maze-engine.js:7065. Builds a shared occupied set keyed "x,y,elevation" (buildOccupiedMap:2093, occupiedElevationKey:518; gems/orange_button/puncher actors are non-blocking and excluded, isNonBlockingType:56-58). Players are processed sequentially via sortPlayersForMove (7072). For each main player (7233+): remove self from occupied (7233), then a step loop (7289): compute target=next+step; resolveIceSlopeTraversal (7376) with pushSlopeBlocker closure (7336-7375, budget=countSupportingPlayers, snapshot+rollback on failure); canEnterHole only when elevation 0 (7434); canPlayerStandAtElevation (7435, requires terrain OR non-main-player actor support and no terrain block, 1692-1717); playerIceSlipLanding (7444, only if the departure cell is ice, 1811); blockingActorAtElevation at target elevation (7460-7469, any non-removed non-collectible actor at exactly that elevation, 2960-2970). Pushing is only allowed on the very first step (isInitialStep = travelPath.length===1 && at start, 7296-7297; canAttemptInitialPush 7479-7480) — a player already sliding on ice stops at a box instead of pushing it. On success continues stepping while landing cell is ice (7786). Final elevation: lift toggle (7831-7838), else slip landing, else hole keeps travelElevation, else playerSurfaceHeightAt snap (7855-7864). Move record pushed at 7937 (after any pushed-box records); gems collected only at endpoint elevation (collectGemsAtEndpoint 7942/3047). Blocked-but-slid players emit visualOnly bounce records (7532-7554, 7957-7974).

Key functions: move:7065, canPlayerStandAtElevation:1692, blockingActorAtElevation:2960, playerIceSlipLanding:1789, findSlideDestination:2655, landingElevationAtLocation:1719

Known quirks/asymmetries noted:
- Push only permitted on the initial step: a player sliding on ice cannot push a box mid-slide (7479-7480) but CAN push a slope blocker mid-slide via pushSlopeBlocker (7336) — asymmetric push windows.
- isInitialStep also fails after options.startOnCurrentSlope traversal moved the player, silently disabling pushes for that whole move.

### Push budget / pushWeightForType

pushWeightForType (maze-engine.js:75-77): box and floating_floor cost 1; every other type (including weightless_box) costs 0. Budget for a main-player push = countSupportingPlayers (2934-2958): 1 + number of player-type actors in a contiguous line directly behind the pusher at the pusher's exact elevation. attemptPushActor (4588) charges cost up-front (4609-4613, returns null if budget < cost), then recursively pushes each blocking pushable with the remaining budget (4781-4803). So 1 player pushes exactly 1 heavy box; a chain of N heavy boxes needs N players lined up behind; any number of weightless boxes ride along free and can be interleaved in a chain. A 'handled' set keyed by pushEntityKey (2150: weightless:<groupId> / clone:<groupId> / actor:<index>) makes a second push of the same entity a free no-op (4605-4607). Clone groups push with budget = max(1, groupMemberCount) regardless of formation (7120).

Key functions: pushWeightForType:75, countSupportingPlayers:2934, attemptPushActor:4588, pushEntityKey:2150

Known quirks/asymmetries noted:
- countSupportingPlayers counts trailing players even if they are themselves blocked, being carried, or standing on a different support — only position+elevation matter.
- Clone-group budget (member count, 7120) vs player budget (trailing line) is a deliberate but undocumented asymmetry.
- Budget compares player elevation only at the pusher's own elevation; a supporting player one level up on the same column contributes nothing.

### attemptPushActor / push chains

attemptPushActor (maze-engine.js:4588-4852). For weightless_box it delegates to collectWeightlessPushCluster + moveWeightlessCluster. For heavy entities (box/floating_floor; members via pushActorMembers:2162) each member's target cell must be: inside board, and either a slope traversal/blocked-slope/fall/top-slide/bounce case, a hole at elevation 0 (4646), or terrain-supported at the member's exact elevation (4737-4744) — actor support (a box below) does NOT count, so heavy boxes can never be pushed onto other actors' tops or off ledges. A blocking actor at the (slope-adjusted) target at the same elevation (4753-4762) must be pushable (4768-4770) else the push fails; pushable blockers are pushed recursively first (4781-4803). Then moveBox (2832) runs findSlideDestination (2655): the box slides step-by-step while landing on ice, requires canMoveInto per step (2795, canMoveInto:2261 = in-board, not terrain-blocked, hole@0 or terrain-supported, not occupied), resolves slope traversals/falls, and finally lands via landingElevationAtLocation (2808; landing can drop several levels and MAY land on actor tops incl. players, 1758). moveBox returns false if the box didn't move at all (2891-2894), which fails the whole push. After the push the pusher must still be able to occupy the vacated/follow target (7675-7727) or everything is rolled back from snapshots (7723-7733). Player follows the pushed box's full ice path only when the player itself stands on ice (canFollowPushedIcePath 7617-7635) and the box path is flat and strictly forward (playerFollowPathForPushedMove 1049-1087); player rides on top when the pushed entity was its own support (pushRidesSupport 7571-7573, playerRidePathForPushedSupport 1144).

Key functions: attemptPushActor:4588, moveBox:2832, findSlideDestination:2655, canMoveInto:2261, playerFollowPathForPushedMove:1049

Known quirks/asymmetries noted:
- Push feasibility uses terrainSupportsElevation only (4737): a heavy box supported by another box cannot be pushed sideways even onto an equal-height actor stack, while weightless boxes can traverse unsupported cells — intended asymmetry but a common author surprise.
- landingElevationAtLocation includes player heads as landing support (includePlayers=true, 1758) while canPlayerStandAtElevation excludes main players (1714) — boxes can rest on players but not vice versa.
- A box sliding on ice stops adjacent to another box (canMoveInto occupied check) — no momentum transfer; chains only push on direct contact at push time.

### Box types M0-M4

M0-M4 are NOT weight tiers. games/maze/level_parsing.json:130-131 maps tokens ["M0".."M4"] to actor type weightless_box; the token index becomes the groupId, so M0-M4 are five independent connection groups of the same weightless (push-cost 0) box type. games/maze/toolbox.json:97-121 documents all five identically: 'A weightless box that can be pushed by anything. All boxes with the same id are connected and move together. They cannot be pushed out of a room.' The heavy 1-weight types are the plain 'box' (crate, token b) and 'floating_floor' (token f) per pushWeightForType:75-77.

Key functions: pushWeightForType:75, level_parsing.json:130

Known quirks/asymmetries noted:
- Task brief said 'boxes M0-M4 differ in weight/behavior' — they don't; only groupId differs. Any observed behavior difference between M-numbers would be a bug.

### Weightless box groups & clusters

Group = all weightless_box actors sharing groupId (weightlessGroupMembers:2191). Members keep fixed relative elevations captured at engine creation (initializeWeightlessRelativeElevations:333-359, base = min member elevation). collectWeightlessPushCluster (3749-3952) expands the pushed group transitively: any weightless box of another group hit in the movement direction at the same elevation joins the cluster (3925-3932); non-weightless pushables become 'blockers' pushed separately (3934-3939); any member whose target is blocked (not slope-traversable, fails canWeightlessMemberEnter with blockSupportSide, and not ice-slope-transit) aborts the whole cluster (3862-3892 push null blocker → return null at 3943). moveWeightlessCluster (4168-4586) then steps the whole cluster rigidly: weightlessClusterStep (3488) requires every member to occupy its target (weightlessMemberCanOccupy:3114 = in-board + not terrain-blocked + not occupied; NO support required), slope-traversing members must all share an identical delta/path (3603-3614); after each step the cluster settles down one layer at a time while no member has support (settleWeightlessClusterDownOneLayer:3293, stops when any member supported or all ≤ 0), and keeps sliding while any member touches ice/hole and none is supported (weightlessClusterShouldContinueSliding:3214-3252). Cluster falls into a hole only when EVERY member is over a hole (4460-4466). weightlessGroupSupportedElevation (1860-1887) computes a group's resting base: max over members of (supportHeight - relativeElevation) for support heights ≤ currentElevation+1 (allows auto step-up of 1), clamped ≥ 0; used by syncWeightlessGroupElevations at engine init (361, called from createInitialState:262-276). Post-move falls use vertical-contact components: weightlessVerticalSupportComponentGroupIds (2232) merges groups whose members stack exactly ±1 on the same cell; weightlessComponentSupportedElevation (1903) + weightlessComponentFallBaseElevation (6659) drop the whole component until supported, all ≤0, or blocked below, removing it (fall into hole) when fully at/below floor with no support (6816-6865). Side-entry rule: terrainSupportSideBlocksWeightlessEntry (731-760) forbids a weightless box entering a cell laterally at elevation == floorOrIceSurfaceHeight-1 (can't clip into the side of a raised floor/ice slab); applied only when actually moving (3098, 3672).

Key functions: collectWeightlessPushCluster:3749, moveWeightlessCluster:4168, weightlessClusterStep:3488, settleWeightlessClusterDownOneLayer:3293, weightlessGroupSupportedElevation:1860, weightlessComponentSupportedElevation:1903, weightlessVerticalSupportComponentGroupIds:2232

Known quirks/asymmetries noted:
- weightlessGroupSupportedElevation clamps base to Math.max(0,...) (1886) but weightlessComponentSupportedElevation has no 0 clamp (1950-1951) — negative bases are possible post-move but not at init.
- The 'support height ≤ currentElevation+1' filter (1878, 1942) silently allows a 1-level auto climb during elevation sync but lateral pushes can never climb (blockSupportSide) — asymmetric climb-up vs push-up.
- Cluster expansion checks blocker elevation equality with the pushing member only (3912); a weightless box of another group offset vertically by 1 in the push direction is treated as an independent blocker (heavy path) or ignored, not merged.
- attemptPushActor collects the weightless cluster once (4617-4629) but moveWeightlessCluster recomputes steps against live state; blockers list order (Array.from of a Set, 3769) is insertion-ordered but cluster iteration mutates while expanding — order-dependent outcomes possible with multiple blockers.

### floating_floor pushing & hole filling

floating_floor is a pushable, weight-1, support-providing actor (isPushableType:60-62, pushWeightForType:75-77, isSupportActorType:64-73). It pushes exactly like a heavy box via attemptPushActor→moveBox. Its unique rule is in applyHoleFalls (6018): when a floating_floor ends its move over a hole at its landing elevation it is removed with skipHoleFall+visibleDuringMove and sets fillsHole/fillHoleX/fillHoleY (6062-6080); applyMoveFinalState then rewrites that terrain cell to floor (6136-6142), permanently filling the hole (undoMove restores the previous terrain via fillHolePreviousTerrain, 8145-8154). Ordinary boxes falling into holes are simply removed (6111-6116) and do NOT fill them. The browser layer mirrors the fill in play-movement.js fillHoleAt (226-233) which replaces the runtime terrain cell with a floor cell.

Key functions: applyHoleFalls:6018, applyMoveFinalState:6122, play-movement.js fillHoleAt:226

Known quirks/asymmetries noted:
- Hole-fill triggers only when the floating_floor is the subject of a move record; a floating_floor that becomes hole-suspended via dynamic support loss goes through ensureDynamicMove/hole-fall paths (6497+) — worth testing that path also fills.

### What blocks a push / a move

Terrain blocking (terrainLayerBlocksElevation:1515-1568): wall/ice_block/block_asset block exactly their layer elevation; ice_slope blocks its elevation and elevation+1 (traversal is the only way through, 1533-1535); shrub blocks [e, e+1] (1537-1539); tree blocks [e, e+3) i.e. 3 levels (1541-1543); player_gate blocks when raised (1545-1547); player_lift blocks when raised (1549-1551); orange_wall blocks its elevation unless all orange buttons are pressed, and a pressed wall can still block one lower as a block (1553-1565). Support (terrainLayerSurfaceHeight:639-675): wall/ice_block/ice_slope/shrub/block_asset top = e+1; tree top = e+3; gate/lift top = e or e+1 by state; orange_wall top = e (pressed) or e+1; floor/ice/others top = e; hole/empty/orange_button give no support. A push additionally fails on: board edge (4732, weightless 3082); insufficient budget (4611); a non-pushable blocking actor at target elevation — players, clones (both not pushable), or any pushable that itself can't move (4768-4770, 4839-4841, 3920-3923); missing terrain support at target for heavy boxes (4737); occupied target for weightless members (3152); side of a floor/ice slab for weightless lateral entry (731-760); mismatched slope deltas within a cluster (3603-3614). Elevation matters everywhere: blockers/occupancy are compared at exact elevation (blockingActorAtElevation:2960, occupied keys:518), so actors at different elevations on the same cell never collide.

Key functions: terrainLayerBlocksElevation:1515, terrainLayerSurfaceHeight:639, terrainSupportSideBlocksWeightlessEntry:731, canMoveInto:2261

Known quirks/asymmetries noted:
- shrub blocks elevations e..e+1 but its surface height is e+1 (652-655) — you can stand on a shrub yet it blocks entry at its top elevation? blocks [e, e+1] inclusive means elevation e+1 entry is blocked even though surface says supported at e+1; standing on shrubs may only be reachable by falling, not walking.
- isWall (1509) uses max surface height > elevation, inconsistent with per-layer blocking used elsewhere; it appears mostly legacy.

### Swap / pass-through rules between actors

There is no swap or pass-through between blocking actors. Gems, orange_button actors and punchers are non-blocking: excluded from the occupied map (2097) and from blocker checks (2967, 4760, 3911) — anything moves through/onto them. All other actors block at their exact elevation only; same cell at different elevation is free. Trailing-follows-leader works because players are processed front-most-first (sortPlayersForMove:7000): the leader's occupied entry is removed (7233) and state position updated before the follower's blockingActorAtElevation check runs against live state, so a column of players moving together compresses correctly. Head-on swaps are impossible: the second mover still sees the first at its (unmoved) cell and players are not pushable, so blockingActorAtElevation !== -1 → break (7562, 7736-7738). Riders standing exactly 1 above a moving clone group / weightless cluster are carried (cloneRidersForMove:1173-1269 validates each rider's forward cell; riders recorded at 4536-4583 and 4116-4163) and then skipped as independent movers via carriedPlayers (7080-7085).

Key functions: sortPlayersForMove:7000, buildOccupiedMap:2093, cloneRidersForMove:1173, blockingActorAtElevation:2960

Known quirks/asymmetries noted:
- Occupied-set staleness: a player's occupied entry is re-added at its FINAL elevation (7977-7982) but intermediate state during another player's turn relies on state arrays, not occupied, for blocker checks — mixing the two sources (occupied for slides/slopes, actorAt for blockers) invites subtle disagreements, e.g. slope traversal checks 'occupied' which still contains not-yet-moved later players.
- For perpendicular ties sortPlayersForMove uses ascending y (dx moves) / ascending x (dy moves) (7023, 7037) — deterministic but means the top/left player always wins contested cells.

### Move ordering & post-pass pipeline

Order within one move() call (7065): (1) sortPlayersForMove (7072) orders live players front-most along the axis: dx>0 → x desc, then y asc, then elevation asc; dx<0 → x asc; dy>0 → y desc, x asc; dy<0 → y asc, x asc (7019-7046); clones use their group's minimum elevation and only one representative per clone group survives the filter (7047-7062). (2) Each player/clone-group resolves fully (including its pushes) before the next; failures roll back via cloneState/occupied snapshots (7226-7230, 7723-7733). Clone groups move as a weightless-style cluster of type 'clone' (7108-7118) with push budget max(1,members) (7120), can ride pushed supports (moveCloneGroupAlongSupportPath:4030) and require at least one supported member after each step (clusterHasSupportedMemberAfterStep:3342, invoked at 4337-4351 for clones only). (3) Post passes (7985-8073): collapseSequentialActorMoves (5041, merges multiple records per actor); syncAttachedPunchersForMoves; applyPunchers, then a sticky-puncher loop bounded by actorCount+width+height (8021-8047); collapse again; applyHoleFalls (6018); pending lift toggles applied (8050-8052); applyMoveFinalState (6122, snaps state to move records); syncDynamicActorElevationsAndFalls (6145, rides actors on gate/lift/orange-wall height changes, recomputes weightless components (6758-6866) and clone groups (6880-6997) with bounded fixpoint loops); puncher + surface-attachment syncs; applyHoleFalls + applyMoveFinalState again. Result: {direction, liftToggles, moved, moves} (8075-8080). moveForSearch (8116) reuses shared snapshot buffers and sets search:true which suppresses visualOnly/iceSlide decoration. undoMove (8126) replays records in reverse, restoring fromX/fromY/fromElevation/fromRemoved and un-filling holes.

Key functions: sortPlayersForMove:7000, collapseSequentialActorMoves:5041, applyHoleFalls:6018, syncDynamicActorElevationsAndFalls:6145, undoMove:8126, moveForSearch:8116

Known quirks/asymmetries noted:
- applyMoveFinalState re-applies each record's toX/toY/toElevation in array order AFTER state was already mutated (8053); if two records for different actors disagree with live state (e.g. dynamic falls appended later), last-write-wins by record order.
- The elevation tiebreak in sortPlayersForMove is ascending (lowest first, 7024) — a player standing on another (stacked) moves before its support, so the support's move can invalidate the rider's landing; riders are only protected when the support is a clone group or weightless cluster (carriedRiders), not when a main player stands on another main player's box.

### Browser layer (play-movement.js)

createMovementController (play-movement.js:4) rebuilds a fresh engine from runtime actor/terrain state on EVERY keypress (createCurrentEngine:34-41, engineActorFromRuntime:22-32 passes explicit elevation so the engine trusts runtime elevations). performPlayerMove (301) clones engine.initialState, calls engine.move with continuePunchSlide/startOnCurrentSlope options (316-319), maps engine records back to runtime actors (moveFromEngineRecord:43-54), snapshots undo state only when moves exist (325-338), records gem collection, computes orange-wall pre-lift move sets for animation staging (preTerrainOrangeButtonLiftMoves:86-130: actors dropping onto elevation 0 orange buttons animate before the wall lowers), then either animates (applyMoveLogicalPositions immediately at 373, lift toggles applied at animation lift-phase 379-384) or applies instantly (applyMoveFinalState:235-288, which also fills holes via fillHoleAt:226 and handles collected-gem ghosting). Undo metadata for ice slides is preserved/reversed (iceSlideMoveMetadata:132, applyUndoIceSlideMetadata:172). previewPlayerMove (409) runs the same engine move without committing runtime state. History capped at 500 (367-369).

Key functions: performPlayerMove:301, createCurrentEngine:34, applyMoveFinalState:235, preTerrainOrangeButtonLiftMoves:86

Known quirks/asymmetries noted:
- applyMoveLogicalPositions (290) sets actor.x/y to toX/toY at animation START, so mid-animation queries (rules, buttons) see final positions while elevation lags until finish — matches gateRenderOverride hack but is a classic source of mid-animation rule glitches.
- engineActorFromRuntime always passes elevation (?? 0), so hasExplicitElevation is always true in browser play — the engine's initial elevation inference (initialActorElevation:304, weightless sync at createInitialState:262-276) is exercised only by tests/solver, a drift risk between authored data and runtime.

## Subsystem: ice

All refs are /Users/jpappas/code/GitHub/PixelGameTest/public/maze-engine.js unless noted. Ice = terrain whose SURFACE height equals the actor's elevation: isIce() (823-844) matches `ice` (surface = layer.elevation, 674) and `ice_block` (surface = elevation+1, 648-656) via terrainLayerOfTypeAtElevation (762-793); ice_slope is NOT ice. Player sliding lives in move()'s per-player while loop (7289-7789): each step resolves slope traversal (7376-7427), hole entry (7434, elevation-0 only), standability (7435), edge slip-off via playerIceSlipLanding (1789-1858, requires the CURRENT cell be icy, 1811), or initial-step-only box push (7479-7480; mid-slide collisions stop the player, test maze-engine.test.js:1475). The slide continues only while the landing cell is ice (7786) and never crosses gems mid-path (collectGemsAtEndpoint 3047, called 7942; test :785). Box sliding uses moveBox (2832-2932) → findSlideDestination (2655-2830): step while canMoveInto (2261, requires support-or-hole) and isIce (2803); slope handling tries resolveIceSlopeTraversal, then blocked-slope push, then fall-onto-slope (2720), then perpendicular top-slide (2733); final elevation from landingElevationAtLocation (1719-1767); iceSlide/iceSlipOff flags 2918-2927. Weightless clusters slide as a unit: weightlessClusterStep (3488-3681, all members must agree on one slope delta, descent delayed for trailing members 3641-3651), loop in moveWeightlessCluster (4236-4483) with per-step gravity settling (settleWeightlessClusterDownOneLayer 3293-3339) and continue-rule weightlessClusterShouldContinueSliding (3214-3252: any non-ice support stops; ice OR HOLE contact continues). ice_slope traversal: entry vectors in iceSlopeTraversalForEntry (860-901) — uphill entry needs elevation === layerElevation, downhill needs layerElevation+1; slope blocks both those elevations to normal movement (1533-1535). resolveIceSlopeTraversal (2292-2401) chains across consecutive slopes and appends fall-traversals on exit; off-board exit is a level exit only with allowLevelExit (2311-2335), passed only for main players (7387, 7090). Blocked slopes: push the blocker at the slope exit (blockedIceSlopePushForEntry 2471-2551) or bounce-and-reverse once if standing on ice (2758-2793 boxes, 7494-7525 players, 3683-3747 weightless; visual-only bob if origin isn't ice, 7527-7555, test :992). Fall-onto-slope requires layer.elevation+1 < fromElevation (2553-2619). After moves, actors on unsupported holes or open pits are removed (6042-6117); floating_floor fills holes (6062-6080); applyMoveFinalState (6122-6143). ice_block is pure static terrain: blocks like a wall at its elevation (1525-1531), slippery on top, not pushable, not meltable (no melt code exists), not climbable except via slopes/falls (test :808 move fails into it).

### Ice detection and surface model

isIce(x,y,elevation) (823-844) is true iff an `ice` or `ice_block` layer's SURFACE height equals elevation, via terrainLayerOfTypeAtElevation (762-793) and terrainLayerSurfaceHeight (639-675): ice surface = layer.elevation (674, default 0), ice_block surface = layer.elevation+1 (648-656). So the top of an ice_block is slippery; ice_slope cells are never 'ice'. isIceOrHole (1357-1362) adds holes (used only by weightless sliding). terrainSideBlockingSupportTypes = {floor, ice} (31-34) side-blocks weightless entry (731-759).

Key functions: isIce (823), terrainLayerOfTypeAtElevation (762), terrainLayerSurfaceHeight (639), isIceOrHole (1357)

Known quirks/asymmetries noted:
- ice_slope excluded from isIce: an actor whose slide deposits it logically 'on' a slope cell never continues sliding by the isIce checks (2751, 2803, 7786); traversal helpers must catch every such case or the slide silently stops

### Player ice-floor sliding

In move() (7065), each non-clone player runs a step loop (7289-7789). A step in direction (stepDx,stepDy) succeeds if: slope traversal resolves (7376-7427), target is an elevation-0 hole (7434), player can stand (canPlayerStandAtElevation 1692), player can slip off an ice edge (playerIceSlipLanding 1789-1858 — only when the CURRENT cell is ice, 1811; lands on highest support strictly below, 1843-1849; keeps elevation and later falls/removes if no support at all, 1851-1857 + 6100-6116), or it is the FIRST step and target actor is pushable (7479-7480). Loop continues only while the new cell is ice at travelElevation (7786) and stops on slip-off (7779) or level exit (7782). Sliding players do NOT push actors mid-slide (blockingActor -> break 7562/7736; test tests/maze-engine.test.js:1475-1493). Gems collected only at the endpoint, and not if the endpoint is a hole (7939-7953, collectGemsAtEndpoint 3047; test :785-806). Lifts toggle only at slide endpoint (7820-7838); gateState/orangeButtonsPressed are computed once per move (7070-7071) so gates/buttons never change mid-slide. options.startOnCurrentSlope (7247-7287) pre-traverses the slope under the player (world-edge transitions from play-gameplay.js); continuePunchSlide (7299-7334) is straight punch momentum ignoring ice.

Key functions: move (7065), playerIceSlipLanding (1789), canPlayerStandAtElevation (1692), collectGemsAtEndpoint (3047)

Known quirks/asymmetries noted:
- Initial-step push budget uses countSupportingPlayers (7340, 7569) but moveBox's own pushSlopeBlocker hardcodes budget 1 (2877) — a slope-exit blocker pushed during a box slide never benefits from player-train strength
- After a successful pushSlopeBlocker push (7408), if the re-resolved traversal is still null the committed push stays in state/moves while the player goes on to break — the box move persists though the player never traversed (revert at 7370-7374 only covers failed attemptPushActor)

### Box (M1/floating_floor) sliding — moveBox/findSlideDestination

moveBox (2832-2932) calls findSlideDestination (2655-2830). ### ice_block terrain

### Slides vs holes, exits, gates, buttons, removal

Holes: enterable mid-slide only at elevation 0 (player 7434, pushed box members 4646, canMoveInto 2270-2272); hole cells are not ice so slides end there; syncMoves marks toRemoved for actors on unsupported holes, and for slides flagged iceSlipOff/punchSlide over open pits (6042-6117, lacksLandingSupport 1769-1787); floating_floor converts the hole to floor instead (6062-6080, applied 6136-6142). Exits: terrainTypes.exit (line 6) is referenced nowhere else in the engine — exit cells behave as plain floor (default surface 674), so a slide stops on an exit like any non-ice cell; win detection lives in the play layer/solver, and slope-driven level exits are a separate mechanism (levelExit records 7912-7917) reserved for main players. Gates/orange walls: raised gates block (1545-1546) and lowered gate surfaces are walkable-not-ice; both gateState and orangeButtonsPressed are frozen at move start (7070-7071) so a slide crossing a button cannot re-raise a wall mid-slide (post-move sync handles it, syncDynamicActorElevationsAndFalls 6145+). Player lifts toggle only at the slide endpoint (7820-7838). Collisions: movement is sequential over sortPlayersForMove order (7072); occupied per-elevation map updated as each actor lands, so a slide sees earlier movers' final positions; slope-exit blockers can be pushed mid-traversal, flat-ice blockers always stop the slider (initial player step excepted).

## Subsystem: devices

### Puncher: trigger conditions

A puncher is a non-blocking actor (isNonBlockingType 56-58) with a facing direction from actor.direction||actor.facing, normalized by normalizePuncherDirection (79-95; anything unrecognized defaults to "right"). applyPunchers (5340-5680) runs after each move. Candidates are actors that have a non-visual, non-removed move record this turn (5515-5533) — i.e. only actors that MOVED can be punched — filtered to players (player/circle_player/clone) and pushables (box/floating_floor/weightless_box) (5535-5540, punchTriggerActorAt 4863-4872). Trigger fires when the candidate occupies the same x,y,elevation as a live puncher (puncherActorAt 4854-4861, checked at 5543-5554). Skipped if the puncher was sticky-attached to that push entity at move start (5556-5567 via puncherWasAttachedToPushEntityAtMoveStart 5203-5222) so a box can't be punched by its own carried fist. Re-trigger guard: triggerKey = pushEntityKey:puncherIndex:x,y,elevation added to a per-move-resolution `triggered` set (5569-5573, 5640) — same entity+puncher+position fires at most once per move(); there is no cross-turn cooldown. Passes loop while new triggers occur, capped at actorCount+width+height (5508). Chained punches work across passes: a punched actor landing on a second puncher is re-punched next pass; passing THROUGH a puncher's cell mid-slide does not trigger (only final positions are checked).

Key functions: applyPunchers (5340), puncherActorAt (4854), punchTriggerActorAt (4863), normalizePuncherDirection (79), puncherWasAttachedToPushEntityAtMoveStart (5203)

Known quirks/asymmetries noted:
- Unknown direction strings silently normalize to "right" (line 94) — a typo'd direction makes a right-facing puncher instead of erroring
- Actors standing on a puncher at move start that don't move never get punched; only movement onto the puncher triggers

### Puncher: punch movement (distance, chains, occupancy)

The punched target plus everything ahead of it forms a train: collectPunchTrainMembers (5259-5316) expands in the punch direction gathering players and pushables at the SAME elevation, adding whole entities (weightless groups / clone groups via pushEntityKey 2150-2160). No weight budget applies — unlike attemptPushActor (4609-4615, cost via pushWeightForType 75-77), punchers move arbitrarily long/heavy trains. All triggered trains move simultaneously one cell per step via moveSimultaneousPunchGroup (5469-5506), stepping until blocked, capped at actorCount+width+height steps (5473). canSimultaneousPunchStep (5407-5467) blocks a step on: board edge (5414), terrainBlocksElevation at the member's current elevation (5418), >1 trains targeting the same cell (5422), a blocking actor outside the train that isn't itself a moving train (5426-5439), head-on swap with another train (5443-5455), or occupied map (5458-5463). Punching into an occupied cell therefore doesn't nudge one square — the train stops adjacent. Elevation never changes during the punch (no slope traversal, no gravity, holes ignored mid-flight); members merge punchSlide:true move records (5643-5656) with punchStart/punchSegments metadata (5068-5125) and the puncher gets a visualOnly punchEffect lunge move (addPuncherVisualMove 5127-5176) — the puncher itself never relocates from punching. Known quirks/asymmetries noted:
- No gravity during or after a punch unless the move is over a hole/pit: a box punched off a ledge at elevation N with floor below stops hovering at elevation N — applyHoleFalls' actorIsOverOpenPit (6100-6109) only removes it if NO support exists at-or-below, and dynamicUnsupportedFallElevation (6269-6333) requires lostDynamicSupportUnderActor (a support that moved), which a self-moved actor doesn't satisfy. ## Subsystem: solver

The solver stack has three search entry points in public/maze-solver.js, all sharing one machinery: SolverHeap (lines 16-97) is a real binary min-heap (bubbleUp/bubbleDown, destructuring swaps) ordered by compareSolverNodes (line 12: priority, then cost, then insertion order). Nodes are pooled plain objects {cost, key, order, path, priority, searchReward, state} (SolverNodePool.acquire 127-139); state buffers are pooled typed-array sextuples (SolverStatePool 99-120 wrapping engine.createStateBuffer at maze-engine.js:412 and copyStateInto at 434 — six TypedArray.set copies, zero alloc after warmup). solveWithAStar (215-351) expands via mutate/undo: for each of 4 directions it calls engine.moveForSearch (maze-engine.js:8116) on current.state in place, computes engine.stateKey on the mutated state (line 316), dedups via bestCostByKey Map<string,number>, clones into a pooled buffer only for improving successors (324), then engine.undoMove (338) reverses using the move records. Duplicate heap entries use lazy deletion (265-269): stale nodes keep their full state buffer and full path string until popped. Heuristic (maze-engine.js:8198-8227) is min Manhattan+elevation distance from any live player to any gem, weight 1 for "astar", 0 for "bfs" (231-234); since ice slides traverse multiple cells per unit-cost move it is inadmissible on ice. findHardestGemPlacement (409-534) and findReachablePositions (577-691) set priority = cost (513, 609, 674), so both are uniform-cost Dijkstra over unit edges — BFS in disguise, where the heap plus order tiebreak reproduces exact FIFO order at O(log n) per op. Paths are stored as full label strings grown by concatenation per successor (332, 487, 672). Progress: reportProgress (200-213) is awaited only every progressYieldStateInterval expansions (default 4096, line 10; workers pass 2048), and workers throttle postMessage to 80ms (world-solver-worker.js:9-23) / 66ms (author-solver-worker.js:17) — negligible overhead. Abort: throwIfSolverAborted (164-172) checks signal per pop and per direction in solveWithAStar/findHardestGemPlacement (262/299, 448/475) but only per pop in findReachablePositions (615); workers get no signal — cancellation is worker.terminate() (world-solver.js resetWorker 208-215). world-solver.js is the main-thread planner: clone() is JSON.parse(JSON.stringify()) (31-33); analysisSignature (198-206) JSON.stringifies full {actors, terrain} then FNV-1a hashes it char-by-char; analysisTargets (301-341) fabricates a target object for EVERY (x,y,elevation) cell plus border exits (16x16x5 room ≈ 1280+320 objects) and ships them via postMessage (analyzeInWorker 244-259). world-solver-worker.js rebuilds a fresh engine and runs a full independent A* per gem (46-64), then one exhaustive findReachablePositions with maxExpandedStates = MAX_SAFE_INTEGER (66-77). Real per-expansion cost is dominated by engine move() plumbing (maze-engine.js:7065-7080): 4x per expansion it allocates an occupied Set with one "x,y,e" template string per blocking actor (buildOccupiedMap 2093, occupiedElevationKey 518), a raised-gate Set with filter arrays (2035), player sort/filter arrays plus a Set (7000-7063), three Int16Array clones (7076-7078), moves/collectedGems/pendingLiftToggles/carriedPlayers containers, a result object and a liftToggles.map array (8075-8080) — roughly 60-100 allocations and ~30-40 short-lived strings per expansion for an 8-actor room, plus up to 4 stateKey strings each doing a full O(cellCount) terrain diff scan (446-476).

### SolverHeap (open list)

True binary min-heap, maze-solver.js:16-97. push/pop with bubbleUp/bubbleDown; comparator compareSolverNodes (line 12) orders by priority, then cost, then order (insertion sequence). Swaps use array-destructuring [a,b]=[b,a] (56-59, 90-93). No decrease-key: duplicates are pushed and stale ones lazily skipped on pop via current.cost !== bestCostByKey.get(current.key) (265-269, 451-455, 617-621).

Key functions: compareSolverNodes (maze-solver.js:12), SolverHeap.push/pop (25-44), bubbleUp (46-62), bubbleDown (64-96)

Known quirks/asymmetries noted:
- For unit-cost searches (priority=cost everywhere except astar) the order tiebreak makes heap order identical to FIFO — a plain queue would be O(1) per op with no comparator or swap allocation
- Destructuring swaps may allocate in unoptimized paths

### Node structure and pooling

Nodes are pooled plain objects {cost, key, order, path, priority, searchReward, state} (SolverNodePool.acquire maze-solver.js:127-139, release 141-149 nulls state and clears path). ### Source of truth verdict for a rewrite

public/maze-engine.js must be the single source of truth for all game rules — every live execution path (browser play via play-movement.js delegation, solvers, author tools, scripts/maze-terminal.js + maze-bridge.js used by the Prime/eval environment, agent runs) already executes moves through it, and docs/prime-verifiers-plan.md:33 declares it canonical. A rewrite should: (1) keep or regenerate the play-core.js rendering-math copies (terrainLayerSurfaceHeight 1904, computeRaisedPlayerGateSet 2287) from the engine, ideally by exporting those primitives from MazeEngine instead of duplicating; (2) replace play-world-transitions.js's private terrain-rule copy (318-404) with engine calls on a temp engine built from neighbor levelState — it is the copy that has already drifted (block_asset); (3) treat games/maze/player.py as legacy to delete/regenerate, never as reference; (4) treat games/maze/level_parsing.json + server/maze-levels.js buildCellStack (339) as the level-format authority since it feeds both parity sides; (5) keep sync-runtime byte-mirroring but close the fails-open gap for newly added files.

Key functions: window.MazeEngine (maze-engine.js:8262), server/maze-levels.js getLevelState (468)

Known quirks/asymmetries noted:
- Cross-level continuation semantics (level exits, punch/ice continuation across rooms) live only in the browser layer, so 'engine as single source of truth' currently has a real hole at world scope — a rewrite needs to pull those rules into the engine or a shared module to be complete
